01 / SCHEMA SURFACE

单文件 vs 散落文件

bytemsg233 的价值，不只是少一个 import。它把“被机器读”和“被人读”这两件事同时压到了同一份输入里。

一个 `.bmsg` 文件负责结构、描述、tag、国际化注释。没有 schema 链路追踪，也没有“先跳去别的文件再理解这里”的心智摩擦。

02 / VARINT

二进制编码原理

Varint 的核心不是“很技术”，而是按数字规模计费。小值只拿走很少的字节，大值才继续扩展。

每字节 7 bit 数据 + 1 bit 是否续写。小数字在大多数业务消息里出现得最频繁，所以整体收益非常稳定。

03 / WIRE TYPES

Wire Type 类型表

ByteMsg 刻意只保留最常用的编码形态，避免把 runtime 复杂度和心智负担一并带到 client 里。

只用 Varint(0) 和 Length-delimited(2)。解析分支更少，生成器约束更明确，SDK 行为更稳定。
Wire Type值含义适用类型格式
Varint 0 可变长度整数 int32 / int64 / uint32 / uint64 / sint32 / sint64 / bool / enum Tag + Varint 值
Length-delimited 2 长度前缀数据 string / bytes / message / list / map Tag + Length + Data
04 / HEADER

Field Header 编码

每个字段的头部都把 tag 和 wire type 压成一个 Varint。这个动作很小，但对整体开销极关键。

公式是 `(tag << 3) | wireType`。tag 1-15 的高频字段基本都能把 header 压在 1 个字节里。
Tag
Wire Type
Field 1 / Varint
(1 << 3) | 0 = 8
Field 2 / String
(2 << 3) | 2 = 18
Field 3 / Bytes
(3 << 3) | 2 = 26
Field 5 / Varint
(5 << 3) | 0 = 40
05 / SIGNED INTS

Zigzag 编码

负数之所以会“膨胀”，不是值本身大，而是普通 Varint 看的是补码。Zigzag 做的是重新映射，而不是魔法压缩。

把 `-1` 映射到 `1`，把 `-2` 映射到 `3`。这样常见负值也能回到 1 字节区间。
原始值Zigzag字节数说明
0010 -> 0
-111-1 -> 1
1211 -> 2
-231-2 -> 3
-214748364842949672955最小 int32 映射到最大 uint32
原始值: 0
Zigzag: 0 (1 字节)
公式: (0 << 1) ^ (0 >> 31) = 0
06 / LENGTH DELIMITED

字符串 / 字节编码

字符串、bytes、嵌套 message、list、map 都走同一类包装：先长度，再内容。统一得足够少，SDK 才能稳定得足够久。

先写长度，后写内容。对 runtime 来说，这意味着边界永远清晰，跳过未知字段也简单。
// "Hello"
05                 // 长度 = 5
48 65 6C 6C 6F     // UTF-8 bytes

// "Hi, 世界!"
0B                 // 长度 = 11
48 69 2C 20 E4 B8 96 E7 95 8C 21
// 子 message 先写入临时 buffer
12                 // field 2, wire type = 2
08                 // 子 message 长度 = 8
08 7B              // field 1: id = 123
12 05              // field 2: len = 5
48 65 6C 6C 6F     // "Hello"
07 / FULL PATH

完整编码走查

把 `{id: 123, name: "Hi"}` 拆到字节层之后，你会发现 ByteMsg 省出来的不是“黑盒优化”，而是每一步都少了一点浪费。

观察每一步的 header、length、content，就能理解为什么字段名不落地到 payload 时会明显变小。
1

字段 1: id = 123

Field Header: (1 << 3) | 0 = 8  -> 08
Value: 123 -> 7B
合计: 08 7B
2

字段 2: name = "Hi"

Field Header: (2 << 3) | 2 = 18 -> 12
Length: 2 -> 02
Content: 48 69
合计: 12 02 48 69
3

最终输出

完整编码: 08 7B 12 02 48 69
总长度: 6 B
对比 JSON: {"id":123,"name":"Hi"} -> 20 B
体积节省: 70%

09 / COLLECTIONS

Map / List 嵌套编码

集合是业务消息里的主角，所以 bytemsg233 把它们直接当一等公民处理，而不是额外拼补丁。

同时，这也决定了对象池 reset 逻辑必须原生懂得怎样把 list / map 清干净。

13 / COMMENTS

国际化演示

双语注释不是展示彩蛋，而是代码生成的一部分。现在 class / struct 注释和字段注释都会一起落到输出里。

`.bmsg` 里写一次 `// "中文" | "English"`，不同语言输出时按 locale 自动切换。

不只是 Go 注释。TypeScript class、Java Javadoc、C# XML 注释、Python 字段说明都会按 locale 生成。

// 用户资料 type UserProfile struct {   // 用户ID   Id uint32 `bytemsg:"1"`   // 用户名   Name string `bytemsg:"2"` }
14 / TOOLING

CLI 使用

初始化、编译、多语言输出、locale 切换，都是同一套 CLI 路径。这个工具链应该是顺手的，而不是“先装三层插件”。

目标不是“看起来功能很多”，而是让你从 schema 到可用 SDK 的路径尽量短。
# 创建新的 schema 文件
$ bytemsg233 init myproject

# 自动生成 myproject.bmsg
schema: bymsg/v1
package: myproject

message Example {
  uint32 id = 1 // "ID" | "ID"
  string name = 2 // "名称" | "Name"
}
# 生成 Go
$ bytemsg233 compile user.bmsg -l go -o ./gen

# 生成多语言，并带对象池辅助代码
$ bytemsg233 compile user.bmsg \
    -l go,csharp,java,typescript,python \
    -o ./gen

# 输出中文注释
$ bytemsg233 compile user.bmsg -l go --locale zh -o ./gen
15 / PRODUCT SHAPE

核心特性

这是 bytemsg233 现在真正的产品轮廓：不只是更小，也更像一套能长期维护的跨语言 schema runtime。

更完整的生成物，意味着更少的二次封装、更少的手写池、更少的注释遗漏。

单文件 .bmsg

一个文件写完定义，不需要 import 链回溯。

紧凑二进制

Varint、Zigzag、压缩 header 组合起来稳定省空间。

多语言输出

Go / C# / Java / TypeScript / Python 同步生成。

原生对象池

每门语言自带 acquire / release / reset 约定。

注释内建

class 注释和字段注释按 locale 原生输出。

Map / List 一等公民

集合不是旁路特性，reset 逻辑直接理解它们。

Agent 友好

输入更少、更平、更容易被自动系统消费。

HTML 独立面

人类展示页自包含，不把 Markdown 当页面依赖。
